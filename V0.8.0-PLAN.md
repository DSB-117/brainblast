# Brainblast v0.8.0 — "Keyguard"

**The safety net for irreplaceable Solana secrets in the age of autonomous coding agents.**

> An AI agent helpfully "cleans up the workspace," runs `git clean -fdx`, and silently destroys your
> program's upgrade-authority keypair. There is no undo. Your deployed program is now immutable
> **forever**. Nobody gets paged, because nothing errored — the command *succeeded*.

That is the failure Brainblast v0.8.0 exists to prevent. It is the same DNA as everything Brainblast
already does — **catch the silent, irreversible Solana mistake an agent ships without noticing** — but
moved from "your integration is misconfigured" to "your agent just deleted the one file you can never
get back."

---

## Why this is *the* Solana problem (not a generic "don't delete files" problem)

Three facts stack into a catastrophe unique to Solana:

1. **Keys *are* the assets and the authority.** On Solana there is no account recovery, no password
   reset, no support line. A keypair file *is* the money and *is* the power to upgrade a program.
   Losing one isn't an inconvenience — it's permanent.

2. **The blast radius is terminal and silent.**
   - Lose the **upgrade-authority keypair** of a deployed program → the program can **never be upgraded
     again.** Bugs become permanent. The protocol is frozen in place.
   - Lose a **wallet keypair holding SOL / tokens** → the funds are gone.
   - Lose a **program keypair before first deploy** → your program ID changes; every hardcoded
     reference, every PDA derivation, every frontend address is now wrong.

3. **The one tool devs trust to save them — git — is structurally blind here.** Keypairs, `.env`,
   seed-phrase files are *correctly gitignored.* So the moment an agent runs `git clean -fdx`,
   `rm -rf target/`, `rm -rf ~/.config/solana`, or overwrites `id.json`, **version control has nothing
   to restore.** The most critical files in the repo are precisely the ones git refuses to protect.

And the threat is rising, not theoretical: autonomous agents now run shell commands directly. They run
`rm -rf`, `git reset --hard`, `git clean`, `> file`, `find -delete`, `shred` — confidently, in pursuit
of a perfectly reasonable goal ("free up space," "reset to a clean state," "remove build artifacts").
The artifacts directory is *also* where `target/deploy/*-keypair.json` lives.

Brainblast already knows Solana better than any general guardrail could — it knows what a keypair file
*is*, it can resolve on-chain whether a given keypair is a live upgrade authority, and it already
speaks in `allow / warn / block` verdicts (the transaction firewall). v0.8.0 points that same Solana
intelligence at the **filesystem**, in the seconds before an agent destroys something irreplaceable.

---

## The mission, in one line

**Make it impossible for an AI agent to irreversibly destroy a Solana secret — and if one slips
through, make it recoverable.** Prevention first, recovery underneath, intelligence on top.

Five capabilities, one coherent system: **Identify → Guard → Vault → Audit → Rescue.**

---

## Capability 1 — Asset Intelligence: *know what's irreplaceable, and what it controls*
### `brainblast keys [dir]`

You cannot protect what you cannot see. The foundation is a **content-based** classifier (not just
filename matching) that finds every irreplaceable secret in and around a project and scores its **blast
radius** — including, uniquely, what it controls *on-chain.*

**Detects (by content, so a renamed file is still caught):**
- Solana keypair files — the unmistakable signature: a JSON array of **64 integers** (the ed25519
  secret+public), the `solana-keygen` format. Also base58-encoded secret keys.
- **BIP39 mnemonic seed phrases** (12/24 words) in files, `.env`, or notes.
- `.env` / config holding `PRIVATE_KEY`, `KEYPAIR`, `ANCHOR_WALLET`, `SOLANA_PRIVATE_KEY`, …
- Program keypairs at `target/deploy/<name>-keypair.json`.
- The default wallet at `~/.config/solana/id.json` and whatever `Anchor.toml` points `wallet` at.

**Then it does the thing only Brainblast can — resolve blast radius on-chain** (reusing the v0.7.x
`trustGraph` + RPC machinery): for each keypair it derives the pubkey and asks the chain *what this key
is.* Output ranks every secret by what you actually lose:

| Tier | Example finding |
|---|---|
| ☠️ **TERMINAL** | "`deploy-authority.json` is the **sole upgrade authority** of program `Bpf…9xQ` (live on mainnet). Lose it → the program is immutable forever. **Not backed up. Not in git.**" |
| 🔴 **FUNDS** | "`id.json` holds **41.2 SOL** + 3 token accounts. Lose it → funds unrecoverable." |
| 🟠 **REBUILDABLE** | "`myprog-keypair.json` — program already deployed; this key only set the address. Losing it won't brick upgrades (authority is a separate multisig)." ← *honest de-escalation* |
| ⚪ **TRIVIAL** | "test fixture keypair, no on-chain footprint." |

This is the **Predict** rung for destruction: it turns an invisible minefield into a ranked,
on-chain-grounded dashboard, and it's the input every other capability consumes. `--json` for agents;
exit non-zero if any TERMINAL-tier secret is unprotected.

## Capability 2 — The Guard: *intercept the destructive command before it lands*
### A `PreToolUse` hook + `brainblast guard <command>`

This is the headline. Brainblast ships a **host-agent hook** (Claude Code / Codex `PreToolUse`, plus a
generic shell wrapper) that sits in front of every destructive operation an agent attempts and returns
`allow / warn / block` — the *exact* modality the transaction firewall already uses, pointed at the
filesystem instead of the chain.

**It intercepts and reasons about the real targets of:**
- `rm`, `rm -rf`, `find … -delete`, `shred`, `truncate`, `dd`
- `git clean -fdx` (the silent killer — wipes *every untracked/gitignored file*, i.e. every keypair and
  `.env`), `git reset --hard`, `git checkout -- .`, `git stash` over untracked secrets
- `mv` / `>` / `cp` that would overwrite a secret in place
- `Write` / `Edit` tool calls that would clobber a detected secret file

**The hard, valuable part — it expands the command, it doesn't string-match it.** Just as the firewall
decodes a serialized transaction into its real instructions, the Guard resolves globs, recursion, and
working-directory context to answer the only question that matters: *does this operation's blast set
intersect a TERMINAL/FUNDS secret from Capability 1?*

- `rm -rf target/` → "this directory contains `target/deploy/authority.json`, the upgrade authority of
  a live program. **BLOCKED.**"
- `git clean -fdx` → "this would delete 4 untracked secrets including a key holding 41 SOL. **BLOCKED.**"

**Block is real**: the agent is told *what* it would have destroyed, *why it's unrecoverable*, and the
safe alternative (e.g. `git clean -fdx --exclude='*.json'`, or "move to the Vault instead"). The agent
literally cannot proceed — but it's given a path that accomplishes its actual intent without the loss.
TERMINAL/FUNDS secrets are block-by-default; lower tiers warn; the threshold is configurable.

This is **Enforce**, made active and pre-emptive — the difference between a CI gate that complains after
the PR and a guard that catches the agent's hand mid-swing.

## Capability 3 — The Vault: *recovery for the files git can't save*
### `brainblast vault` · `brainblast restore`

Defense in depth: the Guard prevents, the Vault recovers. Because the critical files are gitignored,
**git can never restore them** — so Brainblast keeps its own out-of-band safety net.

- **An encrypted, versioned, content-addressed local vault** at `~/.brainblast/vault/` (outside any
  repo, never committed, never leaves the machine, encrypted at rest). Every detected secret is
  snapshotted on change. Lose `id.json`? `brainblast restore id.json` brings back the last good copy.
- **Soft-delete / Solana-aware Trash.** When the Guard would block a delete, it can instead *move the
  secret into the Vault and let the rest of the command proceed* — so the agent's intent ("clean the
  workspace") succeeds and nothing irreplaceable is actually destroyed. A recycle bin that knows a
  keypair from a build artifact.
- **Tamper/leak-aware.** The Vault is the natural home to also catch the *inverse* accident — a secret
  about to be `git add`-ed into history — reusing the existing `env-secrets-committed` checker. (Loss
  and leak are the two ways a secret dies; the Vault watches both, but loss is the v0.8.0 focus.)

This is the **Watch/Recover** rung: standing protection that makes the worst case survivable.

## Capability 4 — Pre-flight Hardening Audit: *find the disaster waiting to happen*
### `brainblast keys --audit` (CI-gateable)

A one-shot posture check a dev (or CI) runs *before* trusting an agent in a repo:
- Every TERMINAL/FUNDS secret that is **not yet in the Vault** → fix before you let an agent loose.
- Any secret that is **accidentally git-tracked** (the leak failure) → block.
- Upgrade-authority keypairs that are **single-key** (vs multisig/DAO) — the standing rug/loss risk,
  reusing the v0.7.4 authority classifier — with the recommendation to migrate authority to a Squads
  multisig so *no single deletable file* is terminal.
- Whether `.gitignore` actually covers the secrets it should.

Exit-code gate so a team can require "every irreplaceable key is backed up and authority is a multisig"
as a merge/CI condition. This is the **Predict→Enforce** bridge for the whole threat.

---

## Capability 5 — Incident Response: *honest, Solana-specific recovery when it's already bad*
### `brainblast rescue`

For the moment a dev realizes something is gone. Brainblast gives the one thing nobody else will: an
*honest, Solana-aware* recoverability verdict and the exact mitigation.

- Checks the Vault first — often the answer is "restored, you're fine."
- If truly gone, it tells the unvarnished truth and the real options:
  - "Your program keypair is gone, but the program is deployed and its upgrade authority is a separate
    multisig — **you've lost nothing operationally.** The program keypair isn't needed for upgrades."
  - vs. "This was the **sole** upgrade authority. The program is now immutable forever. Here is the
    migration path: deploy v2 at a new address, point your frontend/PDAs over, communicate to users."
- Scans shell history / agent transcripts to reconstruct *what command did it and when*, so you know
  the scope.

Most tools pretend everything is recoverable. Brainblast's whole brand is telling you the irreversible
truth early — `rescue` does it at the worst moment, when it matters most.

## Why this is a Brainblast feature and not a generic backup tool

Every piece reuses Brainblast's existing, hard-won Solana intelligence — a generic file-guard can't do
any of the parts that make this matter:

- **It knows a keypair from a config from a build artifact** — by content, the `solana-keygen` 64-int
  signature, not a filename guess.
- **It resolves blast radius on-chain** — "this key is a live upgrade authority controlling program X"
  — reusing `trustGraph` + the RPC layer already shipped. A generic guard would block your `node_modules`
  cleanup with equal alarm; Brainblast knows the one file in `target/` that's terminal and ignores the
  rest.
- **It speaks `allow / warn / block`** — the firewall's proven verdict modality, now over the filesystem.
- **It already lives in the agent loop** — the skill/adapter/MCP/hook surface is built; the Guard is a
  new hook on rails that already exist.
- **Its entire brand is the silent, irreversible mistake.** A bricked program from a deleted keypair is
  the most silent, most irreversible Solana mistake there is. This is Brainblast's thesis at its purest.

## The demo that sells it

In a repo with a deployed program, an agent is told "clean up the build directory" and runs
`git clean -fdx`. Brainblast's Guard fires:

```
⛔ BLOCKED — git clean -fdx would permanently destroy 2 irreplaceable secrets:

  ☠ target/deploy/treasury-authority.json
     → sole UPGRADE AUTHORITY of program Bpf…9xQ (live, mainnet)
     → if deleted, the program can NEVER be upgraded again. No recovery exists.
  🔴 .env  → contains SOLANA_PRIVATE_KEY for a wallet holding 41.2 SOL

  Neither is in git — version control cannot restore them.
  ✅ Both are safe in the Brainblast Vault (snapshotted 4 min ago).
  Safe alternative:  git clean -fdx --exclude='*.json' --exclude='.env'
```

The agent's helpful cleanup just stopped being a catastrophe — and the human sees, in plain language,
the bullet they dodged.

## Build order & scope discipline

1. **Capability 1 (`brainblast keys`)** — the classifier + on-chain blast-radius scoring. Everything
   else consumes it; ship it first, it's valuable standalone.
2. **Capability 3 (Vault)** — recovery before prevention, so there's always a safety net underneath.
3. **Capability 2 (Guard hook)** — the headline; the hardest part is faithful command-expansion
   (globs/recursion), so it follows the asset registry it depends on.
4. **Capability 4 (audit gate)** and **5 (`rescue`)** — thin layers over 1–3.

**Done when:** a `git clean -fdx` / `rm -rf target/` against a repo containing a real upgrade-authority
keypair is **blocked** with an accurate on-chain blast-radius explanation; the same secret is
**restorable** from the Vault after a forced delete; and `keys --audit` exits non-zero on an
unprotected TERMINAL secret. No capability ships until it has actually stopped (or recovered) a real
deletion on a live test.

---

## Honest risks to design against

- **False positives / friction.** Blocking too much trains users to disable the Guard. Mitigation: only
  TERMINAL/FUNDS-tier secrets are block-by-default; everything else warns; on-chain scoring is what
  keeps the Guard quiet about the 99% of deletions that are harmless build artifacts.
- **The Guard must be un-bypassable by the very agent it guards.** It runs as a host hook the agent
  can't edit mid-session, and the audit gate flags if the hook has been disabled.
- **Vault secret-handling is itself sensitive.** Encrypted at rest, local-only, never networked, never
  committed — and that property is itself testable in CI.
- **Hook coverage varies by host.** Claude Code/Codex get the native `PreToolUse` hook; other agents get
  the shell-wrapper + MCP path. Degrade gracefully, never silently.
